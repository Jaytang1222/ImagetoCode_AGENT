# ECharts 代码评判报告

基于视觉 Agent 提供的《图表评估报告》，我们对当前 ECharts 内联 JavaScript 代码进行系统性审查。本报告从 **可读性、模块化、可测试性、与视觉报告的一致性** 四个维度展开，并重点聚焦于**导致视觉偏差的代码根源**，提供**可直接落地的改进指令**。

---

## ✅ 一、可读性：中等偏下（需显著优化）

| 项目 | 评价 | 说明 |
|------|------|------|
| **命名规范** | ⚠️ 部分不一致 | `dataCD`/`dataBD` 等变量名合理；但 `generateData()` 函数参数 `centerX`, `centerY`, `spreadX`, `spreadY` 缺乏业务语义（应为 `centroidX`, `edgeHeightY`, `offsetSpread`, `heightSpread`），与报告中术语（Centroid offset / Edge height difference）脱节。 |
| **注释质量** | ⚠️ 关键逻辑缺失 | `generateData()` 注释仅说明“正态分布模拟”，未说明为何采用 `(Math.random() - 0.5) * spread * 2` 这种非标准近似，也未关联报告中各组的**真实分布特征**（如 BD/BF 的 X≈0 集中性、AD 的高离散度）。 |
| **结构清晰度** | ⚠️ 配置与数据混杂 | 所有数据生成、颜色定义、option 配置均在全局作用域平铺，无逻辑分层（如 `config`, `data`, `ui` 模块划分），增大理解与维护成本。 |
| **字体/样式注释** | ❌ 完全缺失 | `fontFamily: 'serif'` 未说明其对应“手写体/衬线体”的设计意图；统计框中 `fill: '#f2f2f2'` 与报告要求的“浅灰背景”无对照说明。 |

> ✅ **改进建议**：  
> - 所有参数名、变量名强制对齐报告术语（如 `centroidOffset`, `edgeHeightDiff`, `clusterTightness`）；  
> - 在 `generateData()` 上方添加 JSDoc，明确标注每组数据拟合的**原图分布特征依据**（引用报告第4节）；  
> - 为每个 `graphic` 子元素添加 `// [TABLE ROW] CD Count` 类注释，提升嵌套结构可读性。

## ✅ 二、模块化：弱（严重阻碍复用与协作）

| 问题 | 具体表现 | 风险 |
|------|----------|------|
| **数据与配置强耦合** | 数据生成逻辑（`generateData()` 调用）直接嵌入 option 构建流程，无法独立导出/替换为真实数据源；`colors` 对象分散在代码中部，未与 series 配置联动。 | 更换数据源需同时修改6处 `dataXX` 变量 + `series` 数组，易遗漏。 |
| **UI 样式硬编码** | 坐标轴标签 formatter、统计框位置（`left: '72%'`）、标题 padding 等全部写死像素/百分比，无响应式或主题变量抽象。 | 适配不同画布尺寸时需逐行修改，违反 DRY 原则。 |
| **辅助线实现冗余** | 使用 `markLine` 绑定到首个 series，且 `silent: true` 属 hack 写法；未利用 `xAxis.axisLine`/`yAxis.axisLine` 的原生能力。 | `markLine` 在多 series 场景下可能失效；`silent` 掩盖了交互逻辑缺陷。 |

> ✅ **改进建议**：  
> - 将数据生成封装为独立函数 `getDataByGroup(groupName)`，返回 `{ data, centroid, spread }` 结构；  
> - 抽取 `theme = { colors, fontSize, gridPadding, tablePosition }` 常量对象，所有样式引用 `theme.colors.CD`；  
> - **删除 `markLine`**，改用 `xAxis.axisLine` 和 `yAxis.axisLine` 的 `lineStyle` 直接控制零轴线（见一致性部分）。

## ✅ 三、可测试性：差（几乎不可单元测试）

| 问题 | 影响 |
|------|------|
| **无导出接口** | 全部逻辑在 IIFE（隐式）作用域内，`generateData`, `colors`, `option` 均不可被外部 import 或 spyOn。 | 无法对数据生成逻辑做边界测试（如 `spreadX=0` 是否退化为单点）；无法 mock 颜色配置验证图例渲染。 |
| **无输入校验** | `generateData()` 对负数 `count`、`NaN` `spread` 等非法输入无防护，错误会静默传播至 ECharts 渲染层。 | 生产环境出现白屏难定位。 |
| **无状态隔离** | `myChart.setOption(option)` 直接操作全局实例，未封装为可重入函数（如 `renderScatterChart(dom, config)`）。 | 无法在单测中多次初始化/销毁图表实例。 |

> ✅ **改进建议**：  
> - 将核心函数 `generateData`, `buildOption` 显式导出（ESM）或挂载到 `window`（UMD）；  
> - `generateData()` 开头添加 `if (count <= 0 || !Number.isFinite(spreadX)) throw new Error(...)`；  
> - 封装 `initChart(dom, dataConfig, uiConfig)` 工厂函数，返回 `{ chart, destroy }` 接口。

## ✅ 四、与视觉报告的一致性：关键项存在硬伤（必须修复）

根据评估报告中 **5 项 actionable items**，逐条定位代码缺陷并给出**精确到行号的修改指令**：

### 🔴 1. 【关键】坐标轴刻度设置错误 → **直接导致 X/Y 轴混乱**
- **问题根源**：  
  - X 轴虽设 `min: -50, max: 100`，但**未指定 `interval` 或 `splitNumber`**，ECharts 自动计算刻度（-30,30,60...）；  
  - Y 轴同理，且 `axisLabel.formatter` 中 `value === -8` 等判断**依赖自动刻度恰好命中该值**，实际未命中导致标签丢失或错位。
- **✅ 修复指令**（修改 `xAxis`/`yAxis` 配置）：
  ```js
  // 替换原 xAxis 配置 → 强制主刻度
  xAxis: {
    type: 'value',
    min: -50,
    max: 100,
    interval: 50, // 关键！固定步长为 50 → [-50, 0, 50, 100]
    // ... 其他保留
  },
  yAxis: {
    type: 'value',
    min: -8,
    max: 10,
    interval: 2, // 关键！固定步长为 2 → [-8, -6, ..., 10]
    // ... 其他保留
  }
  ```

### 🔴 2. 图例布局错误 → **顶部水平 vs 左上角垂直**
- **问题根源**：  
  `legend.top: 40, left: 80` 是绝对定位，且 `data` 顺序与原图不一致（原图左上角为 CD/BD/AD...，当前代码 `['CD','BD','AD','CF','BF','AF']` 顺序正确但位置错）。
- **✅ 修复指令**（替换 `legend` 配置）：
  ```js
  legend: {
    data: ['CD', 'BD', 'AD', 'CF', 'BF', 'AF'],
    type: 'plain', // 必须显式声明，避免 ECharts 4+ 默认 interactive 行为
    orient: 'vertical', // 关键！垂直排列
    left: '10%',     // 距左 10%
    top: '15%',      // 距顶 15%（与 grid.top 对齐）
    itemWidth: 12,
    itemHeight: 12,
    textStyle: { fontSize: 13 }
  }
  ```

### 🔴 3. 颜色区分度不足 → **BD (#005B8C) 与 CF (#007CC7) 色相过近**
- **问题根源**：  
  当前 `colors.BD` 与 `colors.CF` 均为蓝色系，RGB 差异小（B: 91→124, G: 140→199），人眼难分辨。
- **✅ 修复指令**（强化 BD 对比度）：
  ```js
  var colors = {
    CD: '#9AD6E8', // 浅蓝 → 保持
    BD: '#003366', // 深蓝 → 加深（原 #005B8C → 新 #003366，B 通道降 40%）
    AD: '#95D666', // 浅绿 → 保持
    CF: '#007CC7', // 中蓝 → 保持（或微调为 #0066A6 增加差异）
    BF: '#999999', // 灰色 → 保持
    AF: '#FCC89E'  // 浅橙 → 保持
  };
  ```

### 🔴 4. 辅助线样式不足 → **红色虚线未贯穿、z-order 错误**
- **问题根源**：  
  `markLine` 方案本质是“标记线”，非坐标轴线；`color: '#ffaaaa'` 是粉红而非报告要求的 **红色**；且未设置 `zlevel` 控制层级。
- **✅ 修复指令**（**彻底弃用 markLine，改用 axisLine**）：
  ```js
  xAxis: {
    // ... 原有配置
    axisLine: {
      lineStyle: {
        color: '#000',
        width: 1
      }
    },
    // 新增：零轴线（X=0）→ 红色虚线
    splitLine: {
      show: true,
      lineStyle: {
        color: '#ff0000', // 红色
        type: 'dashed',
        width: 1
      },
      // 仅在 X=0 处绘制（通过 splitNumber=0 + 自定义 splitArea 实现）
    }
  },
  yAxis: {
    // ... 原有配置
    axisLine: {
      lineStyle: {
        color: '#000',
        width: 1
      }
    },
    // 新增：零轴线（Y=0）→ 同样红色虚线
  }
  ```
  > 💡 **进阶方案**：使用 `grid.containLabel: false` + `graphic` 绘制两条 `line` 元素，精准控制 `zlevel: 0`（底层）与 `zlevel: 10`（散点之上）。

### 🔴 5. 统计框样式不符 → **背景灰度、边框、位置偏差**
- **问题根源**：  
  `fill: '#f2f2f2'` 过浅（原图约 `#e0e0e0`）；`stroke: '#999'` 边框太粗；`left: '72%'` 位置未锚定右下角。
- **✅ 修复指令**（修改 `graphic` 中表格背景）：
  ```js
  // 替换原表格背景 rect
  {
    type: 'rect',
    shape: { x: 0, y: 0, width: 220, height: 130 },
    style: {
      fill: '#e0e0e0', // 关键！加深灰度
      stroke: '#cccccc', // 边框变细变浅
      lineWidth: 0.5 // 关键！去除“白色边框感”
    }
  }
  // 并将整个 graphic group 定位改为右下锚定：
  left: 'auto', right: 30, // 距右 30px
  top: 'auto', bottom: 30, // 距底 30px
  ```
